## Gaps / misconceptions (not met or misleading vs PRD)

1. **Реальный API Т-Лизинга**  
   Все требования про **POST**, коды ответов и формализованный контракт (§6 «Интеграция», FR-04) в коде **не реализованы** — только mock. Ошибки API в продакшене не смоделированы (кроме общего `catch`).

2. **«Связь с клиентом» как продуктовая сущность**  
   FR-03 требует явную связку; в данных новой заявки **нет client id / контрагента**, только поля формы. Это соответствует открытому вопросу PRD, но **не закрывает формулировку FR-03** буквально.

3. **Черновик не везде, где может ожидать пользователь**  
   Черновик **сохраняется при ошибке отправки** (API). **Нет автосохранения** при вводе и при уходе со страницы без отправки — риск потери данных при закрытии вкладки до первой неудачной отправки. PRD §4.5 и NFR про черновики при сбое отправки покрыты; **широкий сценарий «не потерять ввод»** — слабее.

4. **Длинный комментарий в истории**  
   В историю пишется `commentPreview` с **обрезкой до 80 символов** при длинном тексте. Для **полного текста заявки** в карточке/аудите это **потеря данных** относительно ожидания «видеть, что отправили».

5. **Демо-строки в истории**  
   `mergeListWithDemoRows` смешивает **встроенные демо-заявки** с реальными из `localStorage`. Для проверки «только мои отправки» это может **вводить в заблуждение** (не описано в PRD).

6. **Статусы заявок (одобрено / отклонено)**  
   Новые пользовательские записи всегда `IN_PROGRESS`. Сценарии APPROVED/REJECTED видны **только на демо**. **Нет обновления статуса с бэкенда** Т-Лизинга.

7. **FR-08 vs формулировка «копировать»**  
   Если требование понимать буквально как **copy-out** из полей формы, текущий UI это **не выполняет** (есть paste-in). Если достаточно **ускорения ввода из канала** — поведение близко по смыслу.

8. **P2 — Уведомления в канале**  
   В PRD отнесено к Коннекту; в приложении **нет** — ожидаемо вне скоупа фронта.

## Conclusion

- **MVP-навигация, форма, валидация, mock-отправка, история с номером заявки, ссылка с параметрами и восстановление черновика после ошибки API** — в целом соответствуют P0 и большей части P1 при чтении PRD как **прототипа**.  
- **Полное соответствие PRD как продакшн-спеки** потребует: реальный POST и контракт, явная политика «связи с клиентом», решение по экрану успеха, сохранению полного комментария, автосохранению черновика (если продукт так решит), и уточнения FR-08 (copy vs paste).
