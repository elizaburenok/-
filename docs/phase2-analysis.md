# Phase 2 — Task analysis (Admin panel concept)

## 1. Scope — screens and modules

| Priority | Module / screen | Purpose |
|----------|-----------------|--------|
| P0 | **Shell / navigation context** | Breadcrumb or nav: «Лизинг» → «Заявки на лизинг»; consistent page background and content width. |
| P0 | **Список заявок (история)** | Таблица или список ранее отправленных заявок (ID партнёра, дата, краткие поля); точка входа «Создать заявку». |
| P0 | **Форма создания заявки** | Поля: ИНН, телефон, комментарий (все обязательны); «Отправить заявку»; валидация с ошибками по полям. |
| P0 | **Результат отправки** | Успех: номер/ID заявки; ошибка API: понятный текст; сохранение черновика при ошибке. |
| P1 | **Диплинк** | Маршрут создания заявки с возможностью префилла из query (например `?inn=&phone=`) для перехода из канала. |
| P1 | **UX ускорения** | Кнопки «Вставить из буфера» или явные поля под копирование для ИНН и телефона. |

Вне концепта админки (но в экосистеме): уведомления Коннект, сам API Т-Лизинга — только контракт и мок на уровне прототипа.

## 2. Complexity flags

| Тема | Риск / неоднозначность |
|------|-------------------------|
| **Связь с клиентом** | Не определено, как по ИНН строится связь в бэкенде — в UI достаточно отправки ИНН и отображения сохранённого ID заявки. |
| **Форматы ИНН/телефона** | Маски и нормализация не заданы — либо мягкая валидация (непустая строка), либо последующее ужесточение. |
| **Черновик** | Не ясно сервер vs клиент — для концепта достаточно `localStorage`. |
| **История** | Нет требований к фильтрам/экспорту — минимальный список. |
| **Figma MCP (desktop)** | `get_design_context` / `get_variable_defs` вернули «select a layer» — извлечение токенов с узла без REST-обхода ограничено; использованы поиск по библиотекам и `get_figma_data` по `fileKey`. |

## 3. Component inventory (expected UI)

- **Навигация / заголовок:** `NavigationBar` (хлебные крошки + заголовок раздела).
- **Первичные действия:** `Button` (Primary — «Создать заявку», «Отправить»; Secondary/Transparent — вторичные).
- **Поля ввода:** текстовое поле с лейблом и ошибкой (**Figma: Text field / Input Field**); многострочное поле для комментария (**Textarea Field**).
- **Список истории:** строки на базе **`Cell`** или компактного списка.
- **Сообщения об ошибке/успехе:** баннер или блок на токенах `primitive.error` / `primitive.success` (отдельный блок или упрощённый alert).
- **Дополнительно:** мелкая кнопка «Вставить из буфера» — **Icon Button / Transparent Button** в терминах Figma TUI.

## 4. Recommended screen structure

```
App
└─ Route: /leasing/applications
   ├─ [Список заявок]  ← NavigationBar (Лизинг › Заявки на лизинг)
   │     ├─ Button «Создать заявку» → /leasing/applications/new
   │     └─ Список (Cell × N) из локального хранилища
   │
   └─ Route: /leasing/applications/new
         ├─ Та же NavigationBar + кнопка «Назад» к списку
         ├─ Форма: TextField (ИНН) + paste | TextField (телефон) + paste | TextArea (комментарий)
         ├─ Button «Отправить заявку»
         └─ После успеха: экран/блок успеха с номером заявки и ссылкой «К списку»
```

**Иерархия вёрстки:** страница `page` (фон `page.secondary`) → контейнер контента `max-width: 720px` (по паттерну списков в Figma Patterns) → вертикальный `column` с `gap` из токенов.
