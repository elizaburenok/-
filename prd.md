# PRD: Кредитная админка — раздел «Лизинг / Заявки на лизинг»

## 1. Product goal & context

### Goal

Дать сотрудникам сопровождения **Т-Лизинга** возможность **оперативно передавать входящие лизинговые заявки в Т-Лизинг от лица клиента**: данные приходят из общего канала (Коннект), сотрудник переносит их в форму в админке и отправляет через интеграцию (POST в API Т-Лизинга).

### Context

- Заявки поступают в **выделенный канал (Коннект)**, где продавцы оставляют запросы.
- Типичное содержимое сообщения: **ИНН**, **номер телефона**, **текстовый комментарий** (запрос клиента, вопросы квалифицированного лида).
- Сотрудник работает в **кредитной админке** в разделе **«Лизинг» → «Заявки на лизинг»**.

---

## 2. User roles and permissions

| Роль | Описание | Ожидаемые права (из требований) |
|------|----------|----------------------------------|
| **Сотрудник сопровождения Т-Лизинга** | Основной пользователь сценария | Доступ к разделу «Лизинг / Заявки на лизинг»: просмотр, создание заявки, отправка в Т-Лизинг, просмотр результата и истории заявок. |
| **Продавец (вне админки)** | Источник заявок в канале | Не описан как пользователь админки; влияет только на входящий поток в Коннект. |

**Примечание:** детальная матрица прав (роли админки, read-only, аудит) в исходных требованиях не задана — см. раздел «Открытые вопросы».

---

## 3. Core features and modules (prioritized)

### P0 — MVP

1. **Навигация:** раздел **«Лизинг» / «Заявки на лизинг»** в кредитной админке.
2. **Создание заявки:** кнопка **«Создать заявку»** → форма с полями:
   - **ИНН** (обязательное)
   - **Номер телефона** (обязательное)
   - **Комментарий с запросом клиента** (обязательное, текст)
3. **Отправка:** кнопка **«Отправить заявку»** → валидация → **POST в API Т-Лизинга** в согласованном формате.
4. **Результат:** экран успеха с **номером заявки**; при ошибке — **понятное сообщение**.
5. **Персистенция:** при успехе — сохранение **ID заявки у партнёра** и **связь с клиентом**; при ошибке — **сохранение черновика**, чтобы не терять данные.
6. **История:** заявка попадает в **историю заявок** в админке после успешной отправки.

### P1 — Удобство и связь с каналом

7. **Быстрый переход:** возможность открыть форму создания **по прямой ссылке** из сообщения в канале.
8. **UX для копирования:** интерфейс формы позволяет **удобно скопировать** ИНН и номер из сообщения для ускорения заполнения (например, отдельные поля, кнопки копирования — уточняется при проектировании UI).

### P2 — Уведомления (упомянуто в сценарии)

9. **Уведомление о новой заявке в канале** — относится к Коннекту; связь с админкой явно не описана (см. открытые вопросы).

## 4. Key user flows

### 4.1 Получение заявки из канала

1. Сотрудник получает уведомление о новой заявке в канале Коннект.
2. В сообщении: ИНН, телефон, комментарий.

### 4.2 Переход в админку

1. Сотрудник открывает кредитную админку.
2. Выбирает **«Лизинг» → «Заявки на лизинг»**.

### 4.3 Создание и отправка (happy path)

1. Нажимает **«Создать заявку»**.
2. Заполняет ИНН, телефон, комментарий (перенос из сообщения).
3. Нажимает **«Отправить заявку»**.
4. Система валидирует обязательные поля → отправляет POST в Т-Лизинг.
5. При успехе: сохраняется ID заявки, связь с клиентом, показывается успех с номером заявки, запись в истории.

### 4.4 Ошибка валидации

1. Пустое обязательное поле + «Отправить» → ошибка с **указанием незаполненного поля**.

### 4.5 Ошибка API / сеть

1. Показывается понятное сообщение об ошибке.
2. **Черновик сохраняется**, чтобы данные не потерялись.

### 4.6 Переход из канала по ссылке

1. Из сообщения в канале — переход по прямой ссылке на форму создания заявки (детали параметров ссылки не заданы).

## 5. Functional requirements

| ID | Требование |
|----|------------|
| FR-01 | Форма создания заявки содержит поля: ИНН, номер телефона, комментарий; все три **обязательны**. |
| FR-02 | По «Отправить» выполняется валидация; при незаполненных полях — ошибки с привязкой к полям. |
| FR-03 | При успешном ответе API сохраняется **ID заявки партнёра** и выполняется **связь с клиентом**. |
| FR-04 | POST-запрос содержит **ИНН, телефон, комментарий** в **согласованном с Т-Лизингом формате**. |
| FR-05 | Успешный сценарий: экран успеха с **номером заявки**; заявка отображается в **истории заявок**. |
| FR-06 | При ошибке отправки: **понятное сообщение** пользователю и **сохранение черновика**. |
| FR-07 | Поддержка **прямой ссылки** на форму создания заявки для перехода из канала. |
| FR-08 | UI формы поддерживает **быстрое копирование** данных из сообщения (ИНН, номер) для ускорения ввода. |

---

## 6. Non-functional requirements

| Категория | Требование / ожидание |
|-----------|------------------------|
| **Надёжность** | Не терять введённые данные при сбое отправки (черновики). |
| **Понятность ошибок** | Сообщения об ошибках API/валидации должны быть actionable и понятны сотруднику. |
| **Интеграция** | Контракт API с Т-Лизингом должен быть формализован (формат полей, коды ответов, идентификатор заявки). |
| **Аудит / история** | История заявок в админке доступна для проверки успешных отправок. |

*Производительность, SLA API, локализация и доступность в исходном тексте не заданы.*

---

## 7. Acceptance criteria (кратко)

- **Обязательные поля:** ИНН, телефон, комментарий; пустые значения при отправке → ошибки по полям.
- **Успех:** корректная форма + успешный ответ Т-Лизинг → экран успеха с номером заявки + запись в истории.
- **Интеграция:** POST с полным набором полей в согласованном формате; ID заявки партнёра сохраняется.
- **Канал:** возможен быстрый переход по ссылке; форма помогает копировать ИНН и телефон.

## 8. Open questions / assumptions

### Открытые вопросы

1. **Аутентификация и права:** нужны ли отдельные роли в админке (только сопровождение vs админ), нужен ли аудит «кто создал заявку»?
2. **«Связь с клиентом»:** как в системе определяется клиент (по ИНН, карточка контрагента, ручной выбор)? Что именно сохраняется в связке?
3. **Формат ИНН и телефона:** маски, длина, нормализация перед отправкой в API?
4. **Прямая ссылка:** нужны ли query-параметры для префилла полей из канала (например `?inn=…&phone=…`)?
5. **Уведомления Коннект ↔ админка:** требуется ли диплинк из конкретного сообщения или достаточно общей ссылки на создание?
6. **Черновик:** где хранится (локально, сервер, сессия), как долго живёт, видимость при повторном входе?
7. **История заявок:** фильтры, экспорт, детальная карточка заявки?
8. **Коды ошибок Т-Лизинга:** маппинг на пользовательские сообщения.

### Допущения (явно не противоречат тексту)

- «Кредитная админка» уже существует; добавляется модуль/раздел «Лизинг».
- API Т-Лизинга и формат POST будут согласованы отдельно; продукт обязан их соблюдать.
- Номер заявки на экране успеха соответствует сохранённому ID/номеру от партнёра.
